## Context

現在の OpenSpec WebUI は `ProjectRegistry` が単一の `activeSession` を持ち、サーバー全体でアクティブプロジェクトが1つだけ存在する。ファイル監視（chokidar）もアクティブプロジェクトに対して1つだけ稼働する。WebSocket `WebSocketManager` は全クライアントに同じメッセージをブロードキャストする。

ユーザーは複数ブラウザタブで別々のプロジェクトを開いて並行作業したいが、現状ではあるタブでプロジェクトを切り替えると全タブが切り替わってしまう。

## Goals / Non-Goals

**Goals:**
- 複数のブラウザタブが同時に異なるプロジェクトを閲覧・操作できる
- 各タブで独立したファイル監視・ライブリフレッシュが機能する
- 既存の単一タブ使いのワークフローに影響しない（後方互換）
- クライアントが切断されたら対応するセッションリソースを解放する

**Non-Goals:**
- 複数サーバーインスタンス間でのセッション共有（単一サーバープロセス前提）
- タブ間のドラッグ＆ドロップやデータ共有
- プロジェクトごとの権限・アクセス制御
- 同一プロジェクトの複数バージョン/ブランチの同時表示

## Decisions

### Decision 1: セッション管理を Map ベースに移行

`activeSession: ActiveProjectSession | null` → `activeSessions: Map<string, ActiveProjectSession>`

**理由**: 最小限の変更で複数セッションをサポートできる。既存の `getActiveProject()` / `getActiveData()` 系メソッドは、後方互換のため "最後にアクティブ化されたプロジェクト"（グローバルデフォルト）を返すようにし、新規に `getSession(projectId)` を追加する。

**代替案**: 各クライアントのコンテキストを独立した `ProjectRegistry` インスタンスに分離 → リソース浪費とレジストリ永続化の競合が複雑化するため却下。

### Decision 2: 参照カウントベースの watcher ライフサイクル

`ActiveProjectSession` に `refCount` を追加。クライアントがプロジェクトにバインドされると `refCount++`、切断・切り替えで `refCount--`。`decrementRef` により `refCount` が正の値から 0 に遷移したときだけ watcher を閉じてセッションを解放する。API on-demand 作成やグローバルデフォルトの事前アクティブ化で生まれた `refCount = 0` セッションは、その後クライアントが参照するまで維持してよい。

**理由**: 単純で予測可能。メモリリークを防ぐ。同時に閲覧しているクライアント数が分かるのでモニタリングにも有用。

**代替案**: TTL ベースの自動破棄 → 設定が複雑でユーザーの意図に反する可能性があるため却下。必要なら後から追加可能。

### Decision 3: WebSocket クライアントに projectId をバインド

`WebSocketManager` 内で `Map<WebSocket, string | null>` を管理する。

- クライアント接続時: `connection:init` で現在のバインド済み projectId（初回はグローバルデフォルト）を送信
- クライアントが `project:bind` を送った時: そのクライアントのバインドだけを更新
- ファイル変更イベント: `ProjectRegistry` の watcher コールバックから `projectId` を伝搬し、該当 projectId にバインドされているクライアントにのみ送信
- `project:switched` イベントを廃止し、代わりにクライアント個別の `project:bound` メッセージを使用

**理由**: イベントルーティングが明確。クライアントは自分が見ているプロジェクトのイベントだけを受信する。

### Decision 4: API エンドポイントは X-Project-Id ヘッダーでプロジェクト指定

既存の `GET /api/project`、`GET /api/specs` などのエンドポイントは、オプショナルな `X-Project-Id` ヘッダーを受け付けるようにする。

- ヘッダーなし → グローバルデフォルト（最後にアクティブ化されたプロジェクト）を返す（後方互換）
- ヘッダーあり → 指定された projectId のデータを返す。sessionId が存在しなければ on-demand で作成
- フロントエンドは現在バインド中の projectId を API 共通レイヤーでヘッダーへ自動注入する

**理由**: 既存のフロントエンドコードへの影響最小。新しいフロントエンドはヘッダー付きでリクエストする。

### Decision 5: グローバルデフォルトプロジェクトの維持

`registry.activeProjectId` は残す。新規クライアント接続時やヘッダーなし API リクエスト時のフォールバックとして機能する。`project:bind` はクライアントローカルな切り替えであり、他クライアントのバインドを変更しない。

## Risks / Trade-offs

- **[リソース消費]** 複数プロジェクトの同時監視で chokidar インスタンスとパース結果がメモリを消費 → **緩和**: 参照カウントで不要セッションは即座に解放。現実的なユースケース（3〜5プロジェクト同時）であれば問題ないレベル
- **[永続化の一貫性]** `projects.json` は引き続き単一ファイル。複数セッションから同時書き込みの可能性 → **緩和**: 既存の `mutationLock`（Promise ベースの排他制御）で保護済み
- **[マイグレーション]** 既存の `activeSession` を使っているコードが多い → **緩和**: グローバルデフォルトアクセサ（`getActiveProject()` 等）を残すので段階的移行が可能
- **[テスト複雑性]** クライアントごとのルーティングのテストケースが増加 → **緩和**: 統合テストでマルチクライアントシナリオを追加
