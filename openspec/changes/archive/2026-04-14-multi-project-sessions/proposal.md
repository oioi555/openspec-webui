## Why

OpenSpec の各工程（探索・設計・実装・検証）は長時間に及ぶため、ユーザーはブラウザの複数タブで並行して別々のプロジェクトを開いて作業する。現状はアクティブプロジェクトがサーバー全体で1つしか存在できず、あるタブでプロジェクトを切り替えると全タブが強制的に切り替わってしまう。これにより並行作業が不可能で、ユーザーのワークフローが阻害されている。

## What Changes

- サーバーが複数プロジェクトを同時にアクティブにできるようにする（プロジェクトごとに独立したセッション、パーサー、ファイル監視を維持）
- WebSocket クライアント（タブ）ごとに「どのプロジェクトを見ているか」をトラッキングし、該当プロジェクトの変更イベントのみを個別に送信する
- プロジェクト切り替え時に、そのタブのコンテキストのみを切り替え、他のタブに影響しないようにする
- 既存の API エンドポイントは後方互換性のためグローバルデフォルトプロジェクトを返しつつ、新しいクライアントは `X-Project-Id` ヘッダーでタブごとのプロジェクトコンテキストを指定できる

## Capabilities

### New Capabilities
- `multi-project-sessions`: 複数プロジェクトの同時アクティブ化、プロジェクトごとの独立したファイル監視・パースセッション管理
- `client-project-binding`: WebSocket クライアント（タブ）とプロジェクトのバインディング、クライアントごとのプロジェクトコンテキスト管理、プロジェクト固有のイベントルーティング

### Modified Capabilities
- `project-registry`: アクティブプロジェクトを1つではなく複数保持できるように要件を変更。セッションのライフサイクル管理（参照カウントベースの watcher/parse 維持）を追加
- `live-refresh`: ブロードキャストからクライアント個別ルーティングへ変更。`project:switched` を全クライアントブロードキャストから該当クライアントのみへの送信に変更。`connection:init` でクライアントが最後に見ていたプロジェクトコンテキストを復元

## Impact

- **サーバー側**: `ProjectRegistry` の `activeSession` を1つから Map<projectId, session> へ移行。watcher とパーサーのリソース管理（参照カウント）が必要
- **WebSocket**: `WebSocketManager` にクライアント→projectId のマッピングを追加。メッセージの送信先をフィルタリング
- **API**: 既存エンドポイントの後方互換性を維持しつつ、`X-Project-Id` ヘッダーでプロジェクトコンテキストを切り替える
- **フロントエンド**: プロジェクト切り替え UI がタブローカルで動作するように変更。WebSocket メッセージ処理の調整
- **リソース**: 複数プロジェクトの同時監視による chokidar インスタンス・メモリ使用量の増加に注意
